#설계 철학

"가볍고, 의존성이 없고, 영원히 깨지지 않는 개인 도구."

BrainBloom은 의도적으로 단일 HTML 파일로 만들어졌다. npm, 번들러, 빌드 파이프라인이 전혀 없다. 브라우저가 <script type="text/babel"> 안의 JSX를 런타임에 직접 변환한다. 이 결정은 다음을 보장한다:

  • 제로 빌드 — 파일 하나를 더블클릭하면 그대로 실행된다. 빌드 환경 깨짐, 의존성 버전 충돌, node_modules 부패 같은 문제가 원천적으로 없다.
  • 완전한 이식성 — 파일 하나가 곧 백업이자 배포 단위다. 어디에 두든 동작한다.
  • 장기 안정성 — 외부 빌드 툴체인에 의존하지 않으므로, 몇 년 뒤에도 같은 방식으로 열린다.
◆
이것은 "몰라서 안 나눈" 것이 아니라 명시적 선택이다. 트레이드오프(파일 크기, 코드 분할의 어려움)는 11장에서 정직하게 다룬다.

#아키텍처 개요

순수 함수 레이어 + React 컴포넌트 레이어의 명확한 분리.

코드는 두 개의 뚜렷한 층으로 구성된다:

① 순수 함수 레이어 (컴포넌트 밖)

React 상태와 무관한 순수 로직 — 파서(parseInput), 트리 순회(walk), 레이아웃 계산(layoutTree), 직렬화(sanitizeTree), 그리고 14개의 Drive API 헬퍼. 이들은 입력→출력이 결정적이라 단위 테스트가 쉽다(8장 참조).

② React 컴포넌트 레이어

메인 App 컴포넌트와 분리된 SettingsModal, NodeView 등. 상태 관리는 useState 44개, 부수효과 useEffect 23개, 그리고 클로저 함정을 피하기 위한 useRef 24개로 구성된다.

✦
순수 로직과 React를 분리한 덕분에, 드라이브 동기화 같은 복잡한 로직을 브라우저 없이 Node 환경에서 시뮬레이션·검증할 수 있었다. 이게 데이터 안전성의 토대다.

#데이터 모델

재귀적 트리 — 내용과 표현(layout)을 분리한 노드 구조.

// 노드 하나의 형태
{
  id, label,                       // 식별자, 텍스트
  children: [ ... ],               // 재귀적 자식 배열
  meta: { date, effort, cost },    // 부가 정보
  color, icons: [ ... ],           // 시각 속성
  pinnedSide, autoSide, collapsed, // 배치·접힘 상태
  _x, _y, _w, _h, _subH            // 레이아웃 캐시 (밑줄 = 휘발성)
}

핵심 설계는 밑줄 접두 필드(_x, _y, …)의 분리다. 이들은 화면 배치 계산 결과를 임시 저장하는 캐시일 뿐, "내용"이 아니다. 저장·비교 시 이 필드들을 제거한 순수 내용만 다룬다 — 이 구분이 7장 변경 감지의 정확성을 만든다.

#텍스트 ↔ 트리 양방향 동기화

왼쪽 텍스트 편집기와 마인드맵이 실시간으로 서로를 반영한다.

사용자는 들여쓰기 기반 텍스트로 트리를 편집할 수도, 노드를 직접 조작할 수도 있다. 양쪽이 같은 데이터를 표현하므로 무한 루프 위험이 존재한다 — 텍스트가 트리를 바꾸고, 그 트리가 다시 텍스트를 바꾸고….

해결: 출처 추적 (source guard)

const syncSourceRef = useRef('text');
// 트리→텍스트 갱신 시 ref를 'tree'로 표시
// 동기화 effect는 출처가 자기 자신이면 건너뛴다 → 순환 차단

여기에 텍스트 편집은 디바운스를 적용해, 타이핑 중 매 키 입력마다 트리를 재구성하지 않는다.

#집중 · 몰입 시스템 — 순수 함수의 재사용

하나의 순수 함수 computeFocusIds가 서로 다른 두 기능을 구동한다.

"편집 중 집중 모드"(편집하는 동안 관련 줄기만 강조)와 "몰입 모드"(보기 전용 — 클릭한 줄기만 강조)는 사용자 입장에선 다른 기능이지만, 본질은 같다: 기준 노드로부터 루트까지의 경로 + 직속 자식을 계산해, 그 집합 밖을 흐리는 것.

computeFocusIds(root, targetId)
// DFS로 루트→target 경로 수집 + target의 직속 자식 합집합 → Set 반환
// target 미발견·null 입력이면 빈 Set (아무것도 강조하지 않음 = 안전한 기본값)

두 기능의 차이는 기준 노드를 누가 정하느냐뿐이다. focusIds useMemo가 모드에 따라 분기한다 — 몰입 모드면 클릭한 노드(immerseFocusId), 편집 집중이면 편집 중 노드(editingId). 흐림 자체는 React가 아닌 CSS 클래스(.dimmed)와 filter가 담당해, 데이터(노드의 저장 색)는 전혀 건드리지 않는다.

흐림 강도는 3단계(dim-soft / dim-medium / dim-strong)로 사용자가 선택하며, React.memo 비교 함수에 isDimmed·focusDimLevel을 포함해 강조 상태 변화가 누락 없이 리렌더되도록 보장했다 — memo 최적화와 새 시각 상태가 충돌하지 않게 하는 것이 이 구조의 핵심 규율이다.

#테마 시스템 — CSS 변수 기반 다크 모드

색을 코드에 흩뿌리지 않고, 변수 한 겹으로 6개 테마를 구동한다.

배경 테마는 밝은 3종(샌드·세이지·스카이)과 어두운 3종(차콜·미드나잇·포레스트). 구현은 <html data-theme="..."> 속성 하나와 CSS 변수 재정의 블록이 전부다.

[data-theme="navy"] {
  --bg:#15213a; --panel:#1f2535; --line:#324269;
  --ink:#e2e9f5; --ink-soft:#98a8c8;            // 글자까지 연동 = 진짜 다크
  --node-input-bg:rgba(36,42,58,.96);            // 편집 입력창도 어둡게
}

핵심 결정 두 가지. 첫째, 다크는 배경만 바꾸는 가짜가 아니라 글자·선·패널·입력창까지 연동했다 — 밝은 배경 전제로 하드코딩돼 있던 색들을 변수(--dot-color, --node-input-bg)로 끌어올리는 선행 작업이 있었기에 가능했다. 둘째, 출시 후 색 조합을 교체할 때 내부 키 이름은 유지하고 색 값만 바꿨다 — 이미 테마를 골라 저장한 사용자의 설정이 깨지지 않는다. 하위 호환은 기능에만 있는 게 아니라 설정값에도 있다.

#구글 드라이브 통합

14개의 헬퍼 함수로 구성된, 권한 최소화 원칙의 동기화 계층.

모든 Drive 헬퍼는 컴포넌트 밖 순수 async 함수이며, 진입 시 ensureDriveToken()을 호출해 토큰 유효성을 보장한다.

함수역할
driveGetOrCreateFolder전용 폴더 BrainBloom 조회·생성
driveSaveNewFilemultipart 업로드, fields로 modifiedTime 회수
driveListFiles수정시각 내림차순 목록
driveLoadFileById파일 본문 로드
driveDeleteFile / driveRenameFile버전 정리용 삭제·이름변경
drivePingabout 엔드포인트로 경량 연결 검사
◆
권한 범위는 drive.file — 이 앱이 만든 파일에만 접근한다. 사용자의 다른 드라이브 데이터는 읽지 못한다. 의도적으로 최소 권한을 택해, 민감 권한 검수 없이도 안전성을 확보했다.

#버전 관리 알고리즘

"날짜별 누적 + 최신 N개 유지"를, 최소 연산으로 구현한 4개의 순수 함수.

파일명 규약: [접두어]YYYY-MM-DD.버전번호. 저장 정책은 4개의 순수 함수로 분리되어 각각 단위 테스트된다:

함수책임
parseDriveFileName파일명 → {접두어, 날짜, 버전} 파싱
maxVersionForDate특정 날짜의 최대 버전번호 산출
filesToDeleteToday오늘 N개 초과분(오래된 것) 선별
planPastCleanup지난 날짜 정리 계획(rename + delete)

왜 "재정렬(A)"이 아니라 "누적(B)"인가

매 저장마다 번호를 1~N으로 재정렬하면 여러 파일을 동시에 rename해야 하고, 중간 실패 시 번호가 꼬인다. 대신 "새 1개 추가 + 오래된 1개 삭제" 방식은 연산이 단순하고 부분 실패에 강하다.

✦
이 선택은 비용이 아니라 견고성을 위한 것이다. API 호출 수를 최소화하고, 각 단계가 독립적이라 한 단계가 실패해도 전체가 깨지지 않는다.

#자동저장 안전장치

코드 리뷰를 통해 식별하고 제거한, 데이터 손상으로 이어질 수 있던 경쟁 조건들.

① 재진입 방지 (re-entrancy guard)

저장이 간격(예: 1분)보다 오래 걸리면, 다음 자동저장 tick이 이전 것과 겹쳐 실행될 수 있다 → 중복 파일·잘못된 삭제. isSavingRef 플래그로 한 시점에 단 하나의 저장만 진행되도록 보장한다(수동·자동 공유).

② 취소 가능한 저장 (cancellation)

자동저장을 끄거나 로그아웃하면, 진행 중이던 저장 파이프라인이 다음 단계 전에 멈춘다:

const runVersionedSave = async (tree, prefix, shouldCancel) => {
  if (cancelled()) return null;  // 목록 조회 후
  // … rename / delete 각 단계마다 재확인 …
  if (cancelled()) return null;  // 저장 직전
  // 단, 저장이 시작된 뒤엔 정리까지 완료 → 상태 일관성 유지
}
✦
핵심 통찰: 취소는 "아무 단계에서나 즉시 중단"이 아니라, "저장 시작 전이면 중단, 시작 후면 일관성을 위해 완료"다. 어설픈 중단이 오히려 파일 상태를 깨뜨릴 수 있음을 인지한 설계다.

③ 변경 감지 (불필요 저장 제거)

자동저장은 내용이 실제로 바뀌었을 때만 동작한다. 레이아웃 캐시(_x 등)를 제거한 직렬화 문자열을 마지막 저장본과 비교 — 좌표만 바뀐 건 "변경"으로 치지 않는다. 이는 무의미한 저장을 없애고, 다중 기기 충돌 가능성까지 낮춘다.

#다중 기기 충돌 감지

"회사에서 방치 → 집에서 작업 → 회사 복귀" 시나리오에서 덮어쓰기를 막는 감지 로직.

각 클라이언트는 자신이 마지막으로 저장/불러온 파일의 modifiedTime(서버 기준)을 lastSyncedTimeRef에 보관한다. 저장 직전과 탭 복귀 시, 드라이브 최신본과 비교한다:

1
탭 복귀 / 저장 직전 — 트리거 발생
2
드라이브 최신 modifiedTime 조회 — 내 기준 시각과 비교
3
최신본이 더 새로우면 → 자동저장 보류 + 사용자에게 안내
4
선택: 최신본 불러오기 / 무시하고 계속 / 보류
◆
비교 기준은 드라이브 서버의 modifiedTime이다. 클라이언트 시계(회사·집 PC의 시각 차이)에 의존하지 않으므로 정확하다. 또한 기준 시각이 없거나(이번 세션 미저장) 내 것이 더 최신이면 묻지 않아, 오탐을 막는다.

#인증 · 토큰 수명 관리

보안과 편의를 모두 잡기 위한, "토큰을 저장하지 않는" 재연결 전략.

  • 토큰은 메모리에만 — 액세스 토큰을 localStorage에 저장하지 않는다(XSS·잔존 노출 위험 회피). 새로고침하면 사라진다.
  • 조용한 재발급 — 토큰 만료 2분 전, ensureDriveToken()이 팝업 없이(prompt:'') 갱신해 1시간 만료로 자동저장이 끊기지 않게 한다.
  • 새로고침 후 자동 재연결 — 토큰이 아니라 "연결한 적 있음"이라는 boolean 플래그만 저장. 재로드 시 이 플래그를 보고 조용히 토큰을 재요청한다. 보안을 지키면서 "연결이 유지된 듯한" 경험을 준다.

연결 상태의 3겹 감시

자동저장이 켜진 동안: 30초마다 토큰 만료 예방 갱신, 10분마다 실제 ping, 탭 복귀 시 즉시 검사. 셋 중 하나라도 실패하면 헤더 "설정" 글자를 빨강으로 — 사용자가 "끊긴 줄 모르고 방치"하는 것을 막는다.

▲
한계 인지: 브라우저는 백그라운드 탭·절전 시 타이머를 늦추거나 멈춘다(정책상 우회 불가). 이를 탭 복귀 즉시 검사로 보완하도록 설계했다.

#공개 서비스 대비 — 보안 계층

"혼자 쓰는 도구"에서 "불특정 다수가 쓰는 서비스"로 — 위협 모델이 바뀌면 방어도 바뀐다.

공급망 방어 — SRI (Subresource Integrity)

CDN에서 받는 React·Babel·jsPDF를 정확한 버전으로 고정하고 sha384 무결성 해시를 부착했다. CDN이 변조돼도 해시가 다르면 브라우저가 로드를 거부한다 — 수천 명이 쓰는 정적 앱에서 CDN은 가장 매력적인 공격 표적이기 때문이다. (구글 GSI는 구글이 버전을 고정하지 않아 SRI가 불가능하며, 이는 CSP의 영역으로 남긴다.)

입력 한도 — DoS 방어

SANITIZE_MAX_DEPTH = 100   // 트리 깊이 (스택 오버플로우 차단)
SANITIZE_MAX_NODES = 20000 // 전체 노드 수 (메모리 고갈 차단)
SANITIZE_MAX_LABEL = 5000  // 라벨 글자 수

악의적으로 깊게 중첩되거나 거대한 JSON을 불러와도 정제 단계에서 잘린다. 한도는 정상 사용 규모(깊이 10 미만, 노드 수백)보다 수십 배 커서, 공격만 막고 사용자는 모르게 동작한다.

같은 브라우저 중복 탭 감지 — BroadcastChannel

같은 브라우저의 두 탭은 IndexedDB를 공유하므로 마지막 저장이 이기는(last-write-wins) 위험이 있다. BroadcastChannel로 who/here 프로토콜을 구성해, 나중에 연 탭에만 경고 배너를 띄운다.

여기엔 시뮬레이션 검증이 잡아낸 버그가 하나 있다 — here 응답은 브로드캐스트라 모든 탭에 퍼지므로, 셋째 탭이 열리면 첫째 탭에도 배너가 뜨는 오류였다. "누군가의 who에 응답한 적이 있으면 나는 기존 탭"이라는 isExistingTab 플래그로 남의 here를 무시하게 해 해결했다. 탭 순서·동시 열림을 포함한 9개 시나리오로 검증.

#방어적 설계 · 복원력

"나쁜 입력이 와도 앱이 죽지 않는다"는 원칙의 구현.

  • 입력 정제 — 불러온 모든 데이터는 sanitizeTree를 거친다. 타입 검증, 중복 id 처리, 알 수 없는 필드 제거. 손상된 파일을 만나도 안전하게 거른다.
  • 경계 방어 — 모든 배열 인덱스 접근(children[idx-1] 등)은 경계 검사 후 수행. 형제 이동·계층 변경에서 범위 초과가 발생하지 않는다.
  • 예외 격리 — JSON.parse, 네트워크 호출은 모두 try/catch로 감싸 사용자 친화적 메시지로 변환. 401/403은 자동 로그아웃 + 재연결 안내로 분기.
  • 로컬 이중 백업 — 드라이브와 별개로 IndexedDB에 디바운스 자동저장 + beforeunload 저장. 네트워크 없이도 작업이 보존된다.
✦
전체 코드 리뷰 결과 — 트리 조작, 인덱스 접근, JSON 파싱, 자동저장 경로 모두 적절히 방어되어 데이터를 잃을 만한 심각한 결함이 없음을 확인했다.

#의도적 트레이드오프

잘 만든 시스템은 한계를 숨기지 않는다 — 알고 선택한 것들.

트레이드오프선택의 이유
단일 파일 (7,898줄)코드 분할의 이점보다 제로 빌드·이식성·장기 안정성이 이 개인 도구에 더 가치 있다. 진짜 불편이 명확해지면 빌드 환경과 함께 재설계할 계획.
가상화 없는 렌더링수백 노드 규모까진 충분히 빠르다. 수천 노드 가상화는 복잡도를 크게 키우므로, 실사용에서 필요해질 때 도입.
이미지 미지원웹 환경에서 이미지를 JSON에 임베드하면 자동저장과 충돌·비대화. 라이트한 도구의 정체성을 지키기 위해 보류.
단일 사용자 동시성저장 시점 충돌 감지로 "기기 전환" 시나리오는 해결. 진정한 동시 협업 편집은 이 도구의 목표가 아니다.
◆
각 항목은 "못 한 것"이 아니라 "이 도구의 목적에 비춰 의식적으로 보류·선택한 것"이다. 트레이드오프를 명시하는 것이야말로 성숙한 엔지니어링의 표식이다.

#검증 방법론

UI 없이 핵심 로직을 검증하는, 3단계 파이프라인.

매 변경은 다음 절차로 검증된다:

1
구문 검증 — JSX를 추출해 @babel/parser로 파싱(SYNTAX OK)
2
변환 검증 — @babel/standalone으로 실제 트랜스파일 확인
3
렌더 검증 — ReactDOMServer.renderToString으로 실제 렌더(흰 화면 결함 사전 차단)

여기에 더해, 순수 로직(버전 관리, 변경 감지, 충돌 판정, 강조 집합 계산)은 Node 환경에서 시나리오별 시뮬레이션으로 검증한다 — 실제 드라이브 호출 없이 "동시 6회 저장 → 1회만 실행", "외부 최신본 판정 5케이스", "React.memo 비교 함수 14케이스", "중복 탭 프로토콜 9시나리오(버그 1건을 출시 전 검출)", "집중·몰입 분기 6케이스" 등을 확인했다.

✦
